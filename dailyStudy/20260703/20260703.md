## 1. 네트워크/AI: Zero-Copy Tensor Streaming 고속 파이프라인

LLM, GPU, NPU 추론 경로에서는 텐서 데이터를 자주 옮기는 것만으로도 병목이 생깁니다. 실무에서는 네트워크 수신 버퍼, 파이프라인 버퍼, 추론 입력 버퍼 사이의 복사 횟수를 줄이는 것이 중요합니다.

핵심 포인트:

- `ReadOnlySpan<float>`를 `ReadOnlySpan<byte>`로 해석해 직렬화 비용을 줄입니다.
- `System.IO.Pipelines`를 사용해 스트리밍 처리 모델을 연습합니다.
- 실제 GPU direct memory 전송은 별도 네이티브 런타임이나 드라이버 API가 필요하지만, 파이프라인 설계는 같은 방향으로 구성할 수 있습니다.

## 2. C# 14 확장 타입 개념: 비관리형 구조체 포인터 슬라이싱

고빈도 네트워크 헤더나 바이너리 프로토콜 DTO는 객체를 계속 만들기보다 구조체의 메모리 표현을 직접 다루는 방식이 유리할 때가 있습니다.

핵심 포인트:

- 포인터 주소를 직접 보관하지 않고, 실습에서는 안전한 바이트 뷰를 통해 개념을 확인합니다.
- C# 14 확장 타입 문법은 실습 환경에 따라 지원되지 않을 수 있어, 현재 실행 가능한 확장 메서드로 대체했습니다.

## 3. C# field 키워드 개념: 초고속 CAS 세미-락 구조

라우팅 분기, 서킷 브레이커, 단일 진입 게이트처럼 아주 짧은 임계 구역에는 무거운 `lock`보다 `Interlocked.CompareExchange` 기반의 CAS 게이트가 적합할 수 있습니다.

핵심 포인트:

- `Interlocked.CompareExchange`는 값이 기대값과 같을 때만 원자적으로 교체합니다.
- `TryEnterRoute()`는 0에서 1로 바꿀 수 있을 때만 진입을 허용합니다.
- 실습 코드는 현재 C#에서 실행되도록 명시적 백킹 필드를 사용했습니다.

실습 코드:

- [3_FieldSemiLock.csx](./3_FieldSemiLock.csx)

## 4. ASP.NET Core HybridCache와 NUMA 파티셔닝 개념

대형 클라우드 인스턴스나 멀티 소켓 서버에서는 CPU 소켓 간 원격 메모리 접근 비용이 성능을 흔들 수 있습니다. 캐시 항목을 논리 파티션으로 나누고, 요청 처리 스레드가 주로 접근하는 파티션을 고정하면 NUMA 친화적인 설계로 확장할 수 있습니다.

핵심 포인트:

- 실제 NUMA 토폴로지 제어는 OS와 런타임 지원이 필요합니다.
- 실습에서는 해시 기반 파티션을 만들어 캐시 항목이 특정 노드 버킷에 배치되는 모델을 확인합니다.
- HybridCache 자체 구현이 아니라, NUMA-aware 캐시 설계 사고를 연습하는 코드입니다.

실습 코드:

## 5. Dynamic PGO와 제네릭 인터페이스 호출 최적화

제네릭 인터페이스 구조를 사용해도 핫패스에서 구체 타입이 안정적으로 반복되면 JIT 최적화가 호출 비용을 줄일 여지가 생깁니다. 설계상 중요한 점은 불필요한 박싱, 반사 호출, 동적 디스패치를 피하고 타입 정보를 컴파일러와 JIT가 이해할 수 있게 유지하는 것입니다.

핵심 포인트:

- 제네릭 제약 조건으로 `TTask : IEngineTask<TData>`를 둡니다.
- `[MethodImpl(MethodImplOptions.AggressiveOptimization)]`로 장기 실행 핫패스 최적화 의도를 표현합니다.
- 실행 결과는 최적화 여부를 직접 증명하지 않고, 호출 경로가 정상 동작함을 확인합니다.

실습 코드:

- [5_PgoGenericsInline.csx](./5_PgoGenericsInline.csx)
