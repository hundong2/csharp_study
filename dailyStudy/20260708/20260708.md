## 1. 소켓 지연 진단과 저오버헤드 타임스탬프

마이크로초 단위 지연을 찾을 때 무거운 애플리케이션 로그를 늘리면 관측 자체가 성능을 바꿀 수 있습니다. 실무에서는 EventPipe, eBPF, LTTng 같은 낮은 오버헤드 진단 도구와 애플리케이션 타임스탬프를 함께 비교합니다.

- 코드: [1_EbpfSocketTracing.csx](./1_EbpfSocketTracing.csx)
- 핵심 문법: `Socket`, `Stopwatch.GetTimestamp`, `IDisposable`, `using`
- 왜 필요한가: 지연 원인이 CLR 스케줄링인지, 커널 소켓 버퍼인지, 네트워크 자체인지 구분해야 대응이 정확합니다.

## 2. 스케줄러 바인딩과 할당 절감

수많은 작은 작업을 처리할 때 작업마다 프록시 객체를 새로 만들면 GC 비용이 커집니다. 실행 가능한 예제에서는 원본 작업 객체를 스케줄러에 직접 넘겨 불필요한 래핑을 피합니다.

- 코드: [2_VirtualThreadExtension.csx](./2_VirtualThreadExtension.csx)
- 핵심 문법: `interface`, `ValueTask`, `await`, 구현 클래스
- 왜 필요한가: 작업 수가 많아질수록 객체 할당과 컨텍스트 전환 비용이 병목이 됩니다.

## 3. CAS 기반 트래픽 게이트

대규모 트래픽 장애 상황에서는 한 번에 하나의 스레드만 장애 방어 모드에 진입해야 할 수 있습니다. `Interlocked.CompareExchange`는 락 없이 이런 진입 게이트를 구현하는 기본 도구입니다.

- 코드: [3_FieldBitwiseGate.csx](./3_FieldBitwiseGate.csx)
- 핵심 문법: `private field`, `property`, `Volatile.Read`, `Interlocked.CompareExchange`
- 왜 필요한가: 장애 시점의 게이트는 빠르고 예측 가능해야 하며, 락 경합을 새 병목으로 만들면 안 됩니다.

## 4. Stripe 기반 캐시 분산

멀티코어 환경에서 모든 요청이 하나의 전역 락이나 하나의 딕셔너리에 몰리면 경합이 생깁니다. Stripe 구조는 키를 여러 구역으로 나누어 경합을 낮추는 기본 설계입니다.

- 코드: [4_NumaStealingCache.csx](./4_NumaStealingCache.csx)
- 핵심 문법: `ConcurrentDictionary`, 배열, 해시 분산, `out` 매개변수
- 왜 필요한가: NUMA 환경에서는 다른 CPU 소켓의 메모리를 자주 건드리는 것도 지연 요인이 됩니다.

## 5. Dynamic PGO와 인터페이스 호출 탈가상화

가독성을 위해 인터페이스를 쓰더라도, 핫패스에서는 가상 호출 비용이 문제가 될 수 있습니다. Dynamic PGO는 실제 실행 프로파일을 보고 자주 쓰는 구현체를 최적화할 수 있습니다.

- 코드: [5_PgoLoopDevirt.csx](./5_PgoLoopDevirt.csx)
- 핵심 문법: `interface`, `sealed class`, 배열, `MethodImplOptions.AggressiveOptimization`
- 왜 필요한가: 추상화를 유지하면서도 JIT가 최적화할 수 있는 단순한 호출 구조를 만드는 것이 중요합니다.

## 정리

- 지연 진단은 애플리케이션 로그만으로 끝나지 않습니다.
- 장애 방어 게이트는 `Interlocked` 기반으로 작고 명확하게 만들 수 있습니다.
- 캐시는 전역 경합을 줄이기 위해 stripe나 파티션 구조를 검토해야 합니다.
- JIT 최적화는 수동 트릭보다 예측 가능한 코드 구조와 측정에서 시작합니다.
