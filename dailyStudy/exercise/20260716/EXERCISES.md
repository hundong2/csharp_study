# 단계별 실습

각 단계가 끝날 때마다 아래 명령을 실행하세요.

```powershell
dotnet run --project .\src\ExpenseApprovalExercise\ExpenseApprovalExercise.csproj -- --self-test
```

실패해도 괜찮습니다. 오류 메시지의 파일명과 줄 번호를 먼저 읽고, 한 번에 한 곳만 수정하세요.

## 1단계: 값과 조건 바꾸기 (10분)

`ExpenseApprovalDemo`의 금액을 `90_000m`으로 바꾸고 실행하세요.

1. 승인 경로가 무엇으로 바뀌는지 예상합니다.
2. 실제 출력을 확인합니다.
3. 숫자 뒤의 `m`을 지웠을 때 왜 `decimal` 인수와 맞지 않는지 컴파일 오류를 읽어봅니다. 확인 후 `m`을 복구하세요.

## 2단계: 새 승인 구간 만들기 (15분)

`ApprovalRoute`에 `FinanceDirector`를 추가하고, 100만원 초과 300만원 이하가 이 경로를 사용하도록 `ApprovalRouter.Decide`를 수정하세요. 현재 장비 한도는 200만원이므로 150만원으로 시험할 수 있습니다.

완료 조건:

- 65만원은 여전히 `TeamLead`입니다.
- 150만원 장비 신청은 `FinanceDirector`입니다.
- 기존 자체 검증이 모두 통과합니다.

## 3단계: 새 파이프라인 규칙 추가 (20분)

설명이 10자보다 짧으면 실패하는 `DescriptionLengthRule : IExpenseRule`을 만드세요.

1. 다른 규칙과 같은 형태로 `Validate`를 구현합니다.
2. `CompositionRoot`의 `rules` 배열에 추가합니다.
3. 자체 검증에 짧은 설명이 실패하는 사례를 하나 추가합니다.

이 단계의 핵심은 `ExpenseService`를 수정하지 않고 새 규칙을 끼울 수 있다는 점입니다. 이것이 개방-폐쇄 원칙의 작은 예입니다.

## 4단계: 실무 확장 생각하기 (말로 설명, 10분)

메모리 저장소를 EF Core 저장소로 바꾼다고 가정하세요.

- 바뀌어야 하는 클래스는 무엇인가요?
- 그대로 둘 수 있는 클래스는 무엇인가요?
- DB 저장이 실패하면 `Result<T>`와 예외 중 무엇으로 처리할지, 이유와 함께 설명하세요.

힌트: 사용자가 고칠 수 있는 입력 오류와 시스템 장애는 성격이 다릅니다.
