## 1. 메모리 매핑 기반 IPC

IPC는 프로세스 간 통신입니다. 서버 하나가 모든 일을 직접 처리하지 않고, 로컬 에이전트, 사이드카, 백그라운드 워커와 일을 나눌 때 필요합니다.

왜 필요한가:

- 같은 머신 안의 프로세스끼리 TCP 포트를 열지 않고 데이터를 주고받을 수 있습니다.
- 큰 데이터를 반복 전달할 때 복사 횟수를 줄이는 설계가 가능합니다.
- 사이드카 구조에서 인증, 로깅, 캐시 예열 같은 부가 기능을 분리하기 좋습니다.

학습 코드:

- [1_SharedMemoryIpc.csx](./1_SharedMemoryIpc.csx)

핵심 문법:

- `using (...)`: 파일 핸들, 공유 메모리 같은 리소스를 자동 정리합니다.
- `byte[]`: 네트워크, 파일, 공유 메모리에서 가장 기본이 되는 바이트 배열입니다.
- `Encoding.UTF8`: 문자열과 바이트 사이를 변환합니다.
- `const`: 실행 중 바뀌지 않는 상수를 선언합니다.

## 2. 구조체 DTO와 할당을 줄인 바이너리 변환

고속 통신에서는 `JsonSerializer`처럼 범용적인 직렬화가 편하지만, 아주 뜨거운 경로에서는 할당과 파싱 비용이 커질 수 있습니다. 이때 구조체의 메모리 표현을 정해진 바이트 버퍼에 쓰는 방식을 사용할 수 있습니다.

왜 필요한가:

- 작은 DTO를 매우 자주 보내는 시스템에서 GC 압박을 줄입니다.
- `Span<byte>`를 사용하면 새 배열을 만들지 않고 기존 버퍼를 재사용할 수 있습니다.
- 바이너리 프로토콜은 빠르지만, 필드 순서와 버전 호환성을 직접 관리해야 합니다.

학습 코드:

- [2_ExtensionZeroCopy.csx](./2_ExtensionZeroCopy.csx)

핵심 문법:

- `record struct`: 값 타입 DTO를 간결하게 정의합니다.
- `[StructLayout(LayoutKind.Sequential)]`: 구조체 필드 배치를 예측 가능하게 만듭니다.
- `Span<byte>`: 복사 없이 메모리 구간을 다룹니다.
- `stackalloc`: 짧게 쓰는 버퍼를 스택에 만듭니다.
- 확장 메서드: `data.WriteTo(buffer)`처럼 기존 타입에 메서드가 있는 것처럼 호출합니다.

## 3. CRDT로 이해하는 다중 리전 병합

글로벌 서비스에서는 서울, 도쿄, 오리건, 프랑크푸르트 같은 여러 리전이 동시에 값을 바꿀 수 있습니다. 모든 업데이트마다 중앙 락을 잡으면 지연 시간이 커집니다. CRDT는 각 리전이 먼저 로컬에서 업데이트하고, 나중에 병합해도 같은 결과에 도달하도록 설계하는 방식입니다.

왜 필요한가:

- 네트워크 지연 때문에 모든 리전을 강하게 동기화하기 어렵습니다.
- 장애 상황에서도 각 리전이 독립적으로 쓰기를 계속할 수 있습니다.
- 병합 규칙이 명확하면 충돌 해결을 애플리케이션 코드에서 예측 가능하게 만들 수 있습니다.

학습 코드:

- [3_HybridCacheCrdt.csx](./3_HybridCacheCrdt.csx)

핵심 문법:

- `Dictionary<TKey,TValue>`: key-value 저장소입니다.
- `foreach`: 컬렉션을 순회합니다.
- 튜플 분해: `(string replica, long value)`처럼 묶인 값을 이름으로 꺼냅니다.
- `Math.Max`: 두 값 중 큰 값을 선택합니다.

## 4. Metrics와 운영 관찰성

성능 개선은 측정 없이는 거의 감입니다. `System.Diagnostics.Metrics`는 .NET 애플리케이션에서 요청 수, 응답 시간, 에러 수 같은 지표를 기록하는 표준 API입니다.

왜 필요한가:

- 장애가 나기 전에 지연 시간 증가나 에러율 상승을 볼 수 있습니다.
- 평균뿐 아니라 p95, p99 같은 꼬리 지연을 추적할 수 있습니다.
- OpenTelemetry와 연결하면 Prometheus, Grafana, OTLP 백엔드로 내보낼 수 있습니다.

학습 코드:

- [4_LockFreeMetrics.csx](./4_LockFreeMetrics.csx)

핵심 문법:

- `Meter`: 메트릭을 만드는 출처입니다.
- `Counter<T>`: 계속 증가하는 값을 기록합니다.
- `Histogram<T>`: 값의 분포를 기록합니다.
- `KeyValuePair`: route, status 같은 태그를 붙일 때 사용합니다.

## 5. UTF-8 검증과 SIMD 폴백 사고방식

API 게이트웨이는 외부 입력을 많이 받습니다. JSON을 파싱하기 전에 바이트가 올바른 UTF-8인지 확인하면, 잘못된 입력을 더 앞단에서 걸러낼 수 있습니다.

왜 필요한가:

- 인코딩이 깨진 입력을 파서나 비즈니스 로직으로 보내지 않습니다.
- 보안 경계에서 입력 검증 책임이 명확해집니다.
- 하드웨어 가속이 가능하면 빠르게 처리하고, 불가능해도 안전한 기본 경로로 동작합니다.

학습 코드:

- [5_Avx512Utf8.csx](./5_Avx512Utf8.csx)

핵심 문법:

- `ReadOnlySpan<byte>`: 복사 없이 읽기 전용 바이트 구간을 전달합니다.
- `Utf8.IsValid`: UTF-8 유효성을 검사합니다.
- `Vector512.IsHardwareAccelerated`: 현재 환경의 SIMD 가속 가능 여부를 확인합니다.
- 배열 리터럴 `[0x7B, 0x22]`: C#의 간결한 컬렉션 표현입니다.

## 실무 아키텍처 메모

IPC, 바이너리 직렬화, CRDT, Metrics, UTF-8 검증은 서로 따로 노는 주제가 아닙니다. 고성능 게이트웨이를 설계한다면 다음 순서로 연결됩니다.

1. 외부 입력은 UTF-8 검증으로 먼저 방어합니다.
2. 내부 핫 패스 DTO는 `Span<byte>` 기반으로 할당을 줄입니다.
3. 같은 노드의 사이드카나 워커와는 IPC로 통신 비용을 낮춥니다.
4. 여러 리전의 상태는 중앙 락보다 병합 가능한 데이터 모델을 우선 검토합니다.
5. 모든 최적화는 Metrics로 측정하고, 실제 병목인지 확인한 뒤 유지합니다.
